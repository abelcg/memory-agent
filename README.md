# Memory Agent — Conversational Agent with Persistent Memory

A conversational AI agent that remembers information across sessions using a hybrid search memory system inspired by the Clawbot architecture. The agent uses **Gemini** for conversation, autonomous memory management, and embeddings via the **Gemini Embedding API** (`gemini-embedding-001`), with **SQLite + FTS5** for persistence.

## Architecture

```
User ──> POST /chat ──> Agent Loop ──> Gemini API (with memory tools)
                                           │
                          ┌────────────────┼────────────────┐
                          v                                 v
                   [memory_search]                   [memory_save]
                          │                                 │
                          v                                 v
                  ┌───────────────┐                ┌───────────────┐
                  │ Hybrid Search │                │  Write to DB  │
                  │               │                │  + Index      │
                  │ vector (0.7)  │                └───────────────┘
                  │ + BM25 (0.3)  │
                  └───────────────┘
                          │
                          v
                   SQLite + FTS5
                   (vectors + full-text)
```

### How It Works

1. **Memory Storage**: Each memory is stored in SQLite with its text content, category, and a 3072-dimensional embedding vector (generated by Gemini's `gemini-embedding-001` API).
2. **Hybrid Search**: When searching, the system combines **vector similarity** (cosine similarity, weight 0.7) with **BM25 full-text search** (via FTS5, weight 0.3) to find the most relevant memories.
3. **Autonomous Memory**: Gemini decides when to search and save memories using function calling. The agent saves preferences, facts, decisions, and context automatically.
4. **Cross-Session Recall**: Memories persist in SQLite across sessions. A new conversation can recall facts from any previous session for the same user.

### Module Breakdown

| File | Purpose |
|------|---------|
| `database.py` | SQLite + FTS5 layer: schema, CRUD, hybrid search algorithm |
| `embeddings.py` | Gemini Embedding API pipeline (3072-dim vectors) |
| `tools.py` | Gemini function declarations (`memory_search`, `memory_save`) and executor |
| `agent.py` | Conversation agent loop with session management |
| `main.py` | FastAPI HTTP API |

## Prerequisites

- **Python 3.11+**
- **[uv](https://docs.astral.sh/uv/)** package manager
- **Google AI API key** — free tier at [aistudio.google.com](https://aistudio.google.com)

## Installation

```bash
# 1. Navigate to the project directory
cd memory-agent

# 2. Create .env file with your API key
cp .env.example .env
```

Edit `.env` and configure:

```env
GOOGLE_API_KEY=your-google-api-key-here
GEMINI_MODEL=gemini-2.5-flash          # or gemini-2.0-flash
```

> **Tip:** Get a free API key at [aistudio.google.com](https://aistudio.google.com). If you hit rate limits with `gemini-2.0-flash`, try `gemini-2.5-flash` which has separate quota.

```bash
# 3. Install dependencies
uv sync
```

## Running Locally

```bash
# Clean any previous database (optional, for a fresh start)
rm -f memory.db

# Start the server
uv run python main.py
```

The server starts on `http://localhost:8000`.

## API Endpoints

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/health` | Health check. Returns `{"status": "ok"}` |
| `POST` | `/chat` | Send a message. Body: `{"user_id", "message", "session_id?"}`. Returns `{"response", "session_id"}` |
| `POST` | `/chat/reset` | Clear a session. Body: `{"session_id"}`. Returns `{"status": "session reset"}` |

## Testing

Open a **second terminal** and run these commands while the server is running:

### Step 1 — Health check

```bash
curl http://localhost:8000/health
# Expected: {"status":"ok"}
```

### Step 2 — Conversation 1: Establish memories (session s1)

```bash
# First message: share preferences
curl -X POST http://localhost:8000/chat \
  -H "Content-Type: application/json" \
  -d '{"user_id": "user1", "message": "Hi! I prefer Python over TypeScript, and I work at Acme Corp as a senior engineer.", "session_id": "s1"}'

# Second message: share project context
curl -X POST http://localhost:8000/chat \
  -H "Content-Type: application/json" \
  -d '{"user_id": "user1", "message": "I am working on migrating our monolith to microservices. We use PostgreSQL and Redis.", "session_id": "s1"}'
```

After these messages, the database should contain memories for: Python preference, Acme Corp role, and microservices migration context.

### Step 3 — Conversation 2: Test cross-session recall (session s2)

```bash
# New session, same user — agent should search memory and recall previous facts
curl -X POST http://localhost:8000/chat \
  -H "Content-Type: application/json" \
  -d '{"user_id": "user1", "message": "Can you recommend a good framework for my project?", "session_id": "s2"}'
```

The agent should reference Python and the microservices migration **without the user restating them**.

### Step 4 — Test other endpoints

```bash
# Reset a session
curl -X POST http://localhost:8000/chat/reset \
  -H "Content-Type: application/json" \
  -d '{"session_id": "s1"}'
# Expected: {"status":"session reset"}

# Auto-generated session_id (omit session_id)
curl -X POST http://localhost:8000/chat \
  -H "Content-Type: application/json" \
  -d '{"user_id": "user1", "message": "What language do I prefer?"}'
# Expected: response references Python, session_id is auto-generated
```

## Deploy to Render

1. Push the `memory-agent/` directory to a GitHub repository.
2. Go to [render.com](https://render.com) and create a **New Web Service**.
3. Connect your repository.
4. Render will auto-detect the `Dockerfile` (or use the `render.yaml` blueprint).
5. Add the environment variables in the Render dashboard:
   - `GOOGLE_API_KEY` — your Google AI API key
   - `GEMINI_MODEL` — `gemini-2.0-flash` (or `gemini-2.5-flash`)
6. Deploy.

> **Note:** On Render free tier, SQLite data persists between requests but is lost on redeploy. For production persistence, consider adding a Render Disk or migrating to PostgreSQL + pgvector.

## Memory Categories

| Category | When to use |
|----------|-------------|
| `preference` | User likes/dislikes, tool choices, style preferences |
| `fact` | Personal facts: name, role, company, team |
| `decision` | Choices made: architecture decisions, tech stack picks |
| `context` | Project context: what they're working on, deadlines |
| `general` | Anything else worth remembering |
